# `.spec`

This folder is the Spec Led Development layer for `jido_memory`.

<!-- covers: spec.workspace.readme_present -->

The goal here is to keep a small current-truth contract for the package close to
the code, tests, and docs that prove it.

## Layout

- `README.md` - this file
- `AGENTS.md` - local operating guidance for agents working in `.spec/`
- `decisions/README.md` - ADR guidance for durable cross-cutting decisions
- `topology.md` - current implemented architecture topology and boundaries
- `specs/spec_system.spec.md` - workspace rules and boundaries
- `specs/package.spec.md` - current package-level behavior
- `specs/provider_architecture.spec.md` - draft provider architecture direction
- `specs/matrix/*.spec.md` - draft matrix subjects for classifying core, optional, and provider-direct memory surfaces
- `state.json` - generated by `mix spec.verify` or `mix spec.check` when the spec tooling is runnable

## Workflow

1. Update current-truth subjects in `.spec/specs/`.
2. Keep planning docs and RFCs outside `.spec/specs/`.
3. Use `.spec/decisions/*.md` only for durable cross-cutting decisions.
4. Tie each requirement to real evidence in code, tests, docs, or commands.
5. Run `mix spec.verify --debug` and `mix spec.check` once dependency resolution is healthy.

## Notes

- `.spec/specs/` is for current truth, not roadmap or phase tracking.
- Supporting design material may live in `docs/`, including `docs/rfcs/`.
- The provider architecture subject is marked draft on purpose so the workspace does not pretend the provider system is already implemented.
